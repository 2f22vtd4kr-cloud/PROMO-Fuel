Task: Harden proxy reconnect logic in Account Factory Step 3

The bug

The factory is wasting fresh SMSPool numbers because Decodo residential proxies drop the SOCKS5 tunnel between Step 2 (connect) and Step 3 (SendCode RPC). When the tunnel drops, Telethon throws "Cannot send requests while disconnected". The current code attempts ONE reconnect on the SAME session-N (same exit node) — if that node is dead, the reconnect also fails, and the factory cancels the number.

We confirmed this is happening on genuinely fresh numbers (contact-check in personal Telegram showed "Not yet on Telegram" for a number the factory cancelled with a ProxyError). The factory never got a SEND_CODE_RESPONSE from Telegram for these numbers — the proxy died first.

File to edit

account_factory.py (repo root)

Background context

The current Step 3 reconnect logic lives around line 2393-2465 in _registration_stream. The flow is:

RawSendCodeRequest fails with "Cannot send requests while disconnected"
Error is classified as transient (_is_transient = True because "disconnected" matches)
ONE reconnect attempt on the SAME proxy_string (same session-N, same exit node)
If reconnect fails → cancel order, buy fresh number
The problem: Decodo session-N is sticky — if the residential peer for session-N is dead, retrying the same session-N will always fail. We need to rotate to a NEW session-N on each reconnect attempt.
