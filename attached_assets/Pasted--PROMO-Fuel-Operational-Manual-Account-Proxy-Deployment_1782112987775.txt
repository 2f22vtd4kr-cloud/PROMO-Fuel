# PROMO-Fuel Operational Manual: Account & Proxy Deployment Pipeline
## CHAPTER 1: Sourcing & Vetting Assets (Avoiding Bad Stock)
Buying Telegram accounts in bulk is an optimization game. If you buy low-quality stock, Telegram’s anti-fraud system will trigger a chain-reaction ban wave.
### How to Spot "Clean" Accounts
When browsing digital goods marketplaces (such as DarkStore, AccsMarket, or private dealer panels), use the following quality matrix to select your stock:
| Metric | High-Quality Stock (Buy) | Low-Quality Stock (Avoid) |
|---|---|---|
| **Format** | .session + .json (Telethon or Pyrogram) | tdata or raw phone numbers |
| **Age (Geomatch)** | 14 to 90+ days old ("Aged" / "Отлежані") | 0 to 3 days old ("Fresh / Автореги") |
| **Security** | 2FA (Two-Factor Password) explicitly included | No 2FA password set |
| **Origin** | Registered on clean physical SIMs or high-grade virtual numbers | Registered on cheap, recycled public VoIP numbers |
> ⚠️ **CRITICAL WARNING:** Never purchase accounts registered using the same data center IP range. Ask the vendor if the accounts were created using residential or mobile proxies. If they were registered on datacenter IPs, they will look compromised the moment you load them.
> 
## CHAPTER 2: The Infrastructure Blueprint
Before adding accounts, you must visualize how data flows. Your Replit server acts as the brains, but the proxy acts as the physical location.
### Traffic Flow Illustration
```
[ Replit Platform Server ]
         │
         ├──► Account #1 Core ──► [ SOCKS5 Proxy A (US IP) ] ──► [ Telegram Server DC 1 ]
         ├──► Account #2 Core ──► [ SOCKS5 Proxy B (UK IP) ] ──► [ Telegram Server DC 2 ]
         └──► Account #3 Core ──► [ SOCKS5 Proxy C (DE IP) ] ──► [ Telegram Server DC 4 ]

```
### Buying the Proxies
 1. Go to a provider (e.g., Smartproxy, Proxy-Seller, or IPRoyal).
 2. Purchase a **Static Residential (ISP)** or **Sticky Mobile SOCKS5** package.
 3. Match the proxy country to the country code of the purchased Telegram accounts (e.g., if the accounts are +7, use a matching regional proxy; if they are +1, use a US proxy).
 4. Export the credentials string. It must look like this: socks5://username:password@ip:port


## CHAPTER 4: Technical Deep Dive & Advanced Architecture
As your setup scales past 100+ accounts, you need to understand what is happening under the hood to prevent performance bottlenecks.
### Anatomy of a .session File
The .session file is actually a localized **SQLite Database** container. When a Python framework like Telethon or Pyrogram initializes, it reads this file to locate key structural details:
 * **Auth Key:** The highly sensitive cryptographic key used to sign messages sent to Telegram.
 * **DC ID:** The specific Telegram Data Center (e.g., DC 1 to DC 5) where the account’s data physically resides.
 * **Server Address / Port:** The exact endpoint your worker establishes connection with.
### The MTProto Handshake Process via SOCKS5
When a background Python worker fires up an account session, it doesn't just connect to the web. It executes a complex sequence over TCP:
```
[Python Worker] ──(Connects via SOCKS5 User/Pass Auth)──► [Proxy Server] ──(Obfuscated MTProto Handshake)──► [Telegram DC]

```
 1. **Proxy Authentication:** The worker establishes an encrypted TCP tunnel with the SOCKS5 proxy server using your string's user and pass.
 2. **IP Obfuscation:** The proxy server masks the Replit origin and routes the outbound payload using its consumer IP space.
 3. **MTProto Transport:** The worker initiates a handshake using Telegram's custom MTProto mobile protocol. If the proxy layer introduces more than 300ms of latency, the handshake drops, causing the worker to mark the account as dead. High-speed residential or ISP proxies are vital here.
### Session Anti-Ban Safe Limits
To mimic human interactions across hundreds of instances, your code relies on structured intervals:
 * **Staggered Connection Delays:** Workers do not log all accounts in at once. They use a randomized stagger window (10 to 30 seconds delay) between accounts to avoid hitting Telegram's rate limiter for concurrent connections from similar locations.
 * **Keep-Alive Pings:** To prevent the SOCKS5 proxy from dropping inactive connections, the system sends an automated background ping to Telegram every 60 seconds to maintain the active session status safely.
# Інструкція користувача PROMO-Fuel: Конвеєр розгортання акаунтів та проксі
## РОЗДІЛ 1: Пошук та перевірка акаунтів (Уникнення неякісного матеріалу)
Купівля акаунтів Telegram оптом — це гра на оптимізацію. Якщо ви придбаєте неякісний матеріал, антифлуд-система Telegram запустить ланцюгову реакцію блокувань, яка миттєво знищить вашу сітку.
### Як визначити "чисті" акаунти
Під час вибору акаунтів на торгових майданчиках (таких як DarkStore, AccsMarket або приватні панелі постачальників), використовуйте таку матрицю якості:
| Метрика | Висока якість (Купувати) | Низька якість (Уникати) |
|---|---|---|
| **Формат** | .session + .json (Telethon або Pyrogram) | tdata або просто номери телефонів |
| **Вік (Відлежка)** | Від 14 до 90+ днів ("Отлежані") | Від 0 до 3 днів ("Автореги") |
| **Безпека** | Двофакторний пароль (2FA) є в комплекті | 2FA пароль відсутній |
| **Походження** | Зареєстровані на фізичні SIM або чисті приватні віртуальні номери | Зареєстровані на дешеві публічні VoIP номери |
> ⚠️ **ВАЖЛИВЕ ПОПЕРЕДЖЕННЯ:** Ніколи не купуйте акаунти, зареєстровані з одного діапазону IP-адрес дата-центрів. Завжди уточнюйте у продавця, чи використовувалися резидентські або мобільні проксі. Якщо реєстрація була на IP дата-центрів, акаунти отримають бан під час першої ж сесії.
> 
## РОЗДІЛ 2: Архітектура інфраструктури
Перед додаванням акаунтів необхідно чітко уявити, як рухаються дані. Ваш сервер на Replit виконує роль "мозку", а проксі визначає фізичне розташування акаунта.
### Схема розподілу трафіку
```
[ Сервер платформи Replit ]
         │
         ├──► Ядро акаунта #1 ──► [ SOCKS5 Проксі A (IP США) ] ──► [ Дата-центр Telegram DC 1 ]
         ├──► Ядро акаунта #2 ──► [ SOCKS5 Проксі B (IP Англії) ] ──► [ Дата-центр Telegram DC 2 ]
         └──► Ядро акаунта #3 ──► [ SOCKS5 Проксі C (IP Німеччини) ] ──► [ Дата-центр Telegram DC 4 ]

```
### Купівля проксі
 1. Перейдіть на перевірений сервіс (наприклад, Smartproxy, Proxy-Seller або IPRoyal).
 2. Придбайте пакет **Статичних резидентських (ISP)** або **Sticky Мобільних SOCKS5** проксі.
 3. Обов'язково підбирайте країну проксі під країну куплених акаунтів (якщо акаунти +7 — беріть відповідний регіон, якщо +1 — проксі США).
 4. Експортуйте рядок доступу. Він повинен мати такий вигляд: socks5://username:password@ip:port

## РОЗДІЛ 4: Глибокий технічний аналіз та архітектура
Коли ваша мережа перевищить 100+ акаунтів, вам знадобиться розуміння внутрішніх процесів для запобігання втрати швидкості та блокувань.
### Структура файлу .session
Файл .session — це локалізована **база даних SQLite**. Коли Python-фреймворки (Telethon або Pyrogram) ініціалізують акаунт, вони зчитують з цього файлу такі критичні параметри:
 * **Auth Key (Ключ авторизації):** Зашифрований криптографічний ключ, яким підписується кожен запит до серверів Telegram.
 * **DC ID:** Номер конкретного дата-центру Telegram (від DC 1 до DC 5), де фізично зберігаються дані цього акаунта.
 * **Server Address / Port:** Точна IP-адреса сервера Telegram для встановлення з'єднання.
### Процес рукостискання (Handshake) MTProto через SOCKS5
Коли фоновий воркер активує сесію акаунта, процес не просто надсилає запит в інтернет. Він виконує чітку послідовність дій через TCP-протокол:
```
[Python Воркер] ──(З'єднання з авторизацією проксі)──► [Проксі-сервер] ──(Зашифрований MTProto Handshake)──► [Дата-центр TG]

```
 1. **Авторизація на проксі:** Воркер підключається до SOCKS5 сервера, передаючи логін (user) та пароль (pass) через зашифрований TCP-тунель.
 2. **Маскування IP:** Проксі-сервер приховує реальний IP-адрес Replit і перенаправляє вихідний трафік через пул адрес звичайних провайдерів інтернету.
 3. **Транспорт MTProto:** Воркер починає сесію, використовуючи мобільний протокол MTProto. Якщо затримка (ping) проксі перевищує 300мс, з'єднання розривається, і система маркує акаунт як неактивний. Тому швидкість резидентських проксі є вирішальним фактором.
### Технічні ліміти для захисту від банів
Щоб дії сотень акаунтів виглядали як поведінка реальних людей, платформа використовує вбудовані ліміти:
 * **Почергова ініціалізація (Staggered Delay):** Воркери не запускають всі акаунти одночасно. Використовується випадкова затримка (від 10 до 30 секунд) між підключеннями, щоб Telegram не зафіксував масовий синхронний вхід з одного регіону.
 * **Keep-Alive Пінги:** Щоб проксі-сервер не закривав тунель через відсутність активності, система кожні 60 секунд надсилає фоновий мікрозапит (пінг) в Telegram, підтримуючи статус сесії в безпечному робочому стані.
